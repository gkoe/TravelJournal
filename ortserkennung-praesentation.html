Was ist Reverse Geocoding?

Normale Suche (Geocoding): „Wien" → 48.2089, 16.3725

Reverse Geocoding: 48.2089, 16.3725 → „Wien"

Wozu brauchen wir das?

GPS-Fotos der Radtour enthalten nur Koordinaten. Der Ortsname soll in der Diashow als Beschriftung erscheinen — automatisch, ohne manuelle Eingabe.

Kostenloser Dienst

OpenStreetMap bietet den Nominatim-API kostenlos an — allerdings mit Nutzungsbedingungen: max. 1 Request/Sekunde und User-Agent-Pflicht.

EXIF-GPS-Daten aus dem Foto lesen

Was ist EXIF?

JPEG-Dateien enthalten einen versteckten Metadaten-Block (EXIF = Exchangeable Image File Format). Smartphones schreiben dort automatisch GPS-Koordinaten rein.

"GPS IFD": {
  "GPS Latitude Ref": "N",
  "GPS Latitude": "48° 12' 32.04"",
  "GPS Longitude Ref": "E",
  "GPS Longitude": "16° 22' 21.00"",
  "GPS Altitude": "215 m"
}

Format: Grad/Minuten/Sekunden (DMS) → wird von der Bibliothek automatisch in Dezimalgrad umgerechnet.

ExifReaderService.cs

var directories =
    ImageMetadataReader.ReadMetadata(filePath);

// GPS-Verzeichnis suchen
var gps = directories
    .OfType<GpsDirectory>()
    .FirstOrDefault();

// Koordinaten als Dezimalgrad holen
var loc = gps.GetGeoLocation();
if (loc != null)
{
    photo.Latitude  = loc.Value.Latitude;
    photo.Longitude = loc.Value.Longitude;
}
MetadataExtractor (NuGet) liest alle EXIF/IPTC/XMP-Tags und konvertiert DMS → Dezimal intern — kein eigener Umrechnungscode nötig.

JSON parsen & Ortsname extrahieren

Priorisierungs-Kaskade

Nicht jeder Ort hat alle Felder — wir nehmen den ersten verfügbaren Wert:

  • 1
    address.village Ideal für ländliche Orte der Radtour
  • 2
    address.town Mittelgroße Städte
  • 3
    address.city Großstädte
  • 4
    address.municipality Verwaltungseinheit als Fallback
  • 5
    name / display_name Letzter Ausweg (erstes Komma-Segment)

ParseLocation() in C#

private static string? ParseLocation(string json)
{
    using var doc = JsonDocument.Parse(json);
    var root = doc.RootElement;

    // Hilfsfunktionen
    string? GetAddr(string name) { ... }
    string? GetProp(string name) { ... }

    var place = GetAddr("village")
             ?? GetAddr("town")
             ?? GetAddr("city")
             ?? GetAddr("municipality")
             ?? GetProp("name")
             ?? GetProp("display_name")
                 ?.Split(',')
                 .FirstOrDefault()?.Trim();

    if (place == null) return null;

    // Stadtteil ergänzen wenn vorhanden
    var suburb = GetAddr("suburb");
    return suburb != null
        ? $"{place}, {suburb}"
        : place;
}

Rate-Limiting — max. 1 Request/Sekunde

Nominatim ist kostenlos, aber darf nicht überlastet werden. Die Policy: maximal 1 Request pro Sekunde und App. Wir halten das mit einem Semaphor ein.

Was ist ein Semaphor?

SemaphoreSlim(1,1) — wie eine Tür, durch die nur 1 gleichzeitig darf
WaitAsync() — Eingang sperren. Alle weiteren warten, bis Freigabe
Release() — Eingang freigeben, erst nach 1,1 Sekunden Pause
Warum 1,1 Sekunden?
1,0 wäre knapp — bei Systemlast könnten wir leicht drüber sein. 100 ms Puffer genügen, um sicher unter der 1/s-Grenze zu bleiben.

Der Trick: Delay im Background

public async Task<string?> ResolveAsync(...)
{
    // 1. Eingang sperren
    await _rateLimiter.WaitAsync(ct);
    try
    {
        // 2. HTTP-Request
        var response = await _httpClient
            .GetAsync(url, ct);
        return ParseLocation(...);
    }
    finally
    {
        // 3. Eingang ERST nach 1,1s freigeben
        _ = Task.Delay(1100, CancellationToken.None)
                .ContinueWith(_ =>
                    _rateLimiter.Release());
    }
}

CancellationToken.None beim Delay — selbst bei Abbruch soll die Freigabe nach 1,1s garantiert erfolgen.

Async / Await & Abbruch

Warum async?

Der HTTP-Request dauert ggf. mehrere hundert Millisekunden. async/await gibt den UI-Thread in dieser Zeit frei — das Fenster bleibt responsiv.

Ohne async: UI friert ein, Nutzer denkt, Programm hängt
Mit async: UI läuft weiter, Fortschritt sichtbar in StatusBar

CancellationToken — Abbruch

Der Nutzer kann die Geocodierung jederzeit abbrechen. Das Signal wird über einen CancellationToken weitergegeben.

Toggle-Pattern im ViewModel

private CancellationTokenSource? _cts;

private async Task ToggleGeocodingAsync()
{
    // 2. Klick = Abbruch
    if (IsGeocodingRunning)
    {
        _cts?.Cancel();
        return;
    }

    _cts = new CancellationTokenSource();
    IsGeocodingRunning = true;
    try
    {
        foreach (var photo in queue)
        {
            _cts.Token.ThrowIfCancellationRequested();

            var loc = await _geocoder
                .ResolveAsync(photo.Lat,
                               photo.Lon,
                               _cts.Token);
        }
    }
    catch (OperationCanceledException)
    {
        BackgroundActivityText = "Abgebrochen.";
    }
    finally { IsGeocodingRunning = false; }
}

Cache — Decorator Pattern

Schon abgefragte Koordinaten sollen nicht nochmal zum Server geschickt werden — das spart Zeit und schont die Nominatim-Server.

OUTER

→
INNER

Interface-Trick

// Beide implementieren dasselbe Interface
public interface IReverseGeocoder
{
    Task<string?> ResolveAsync(
        double lat,
        double lon,
        CancellationToken ct);
}

// Aufbau in der Factory:
return new JsonGeocodeCache(
    new NominatimGeocoder(_httpClient),
    cachePath
);

Der Rest des Codes kennt nur IReverseGeocoder — ob Cache oder direkter Request, ist ihm egal.

Cache-Schlüssel & Speicherung

// Schlüssel: 4 Dezimalstellen ≈ 11 m Genauigkeit
var key = $"{Math.Round(lat, 4)}"
        + $",{Math.Round(lon, 4)}";

// Lookup
if (_cache.TryGetValue(key, out var cached))
    return cached;   // sofort zurück

// Neues Ergebnis → speichern
_cache[key] = result;
SaveCache();       // JSON atomar schreiben
Atomares Schreiben: erst in .tmp-Datei, dann File.Move(..., overwrite:true) — verhindert korrupte JSON-Datei bei Absturz.

IsBusy NICHT gesetzt

Geocoding blockiert NICHT alle anderen Aktionen — der Nutzer kann weiterscrollen, filtern, exportieren.

✗ IsBusy = true würde UI einfrieren
✓ IsGeocodingRunning nur für Button-Farbe
💡
Entwurfsentscheidung: Geocodierung verarbeitet nur Selected-Fotos — Fotos, die der Nutzer explizit in die Diashow aufgenommen hat. „Offene" und abgewählte Fotos werden übersprungen.

Testbarkeit

Tests sollen keine echten HTTP-Requests machen — das wäre langsam, flakig und würde Nominatim belasten.

Fake-Geocoder für Tests

private sealed class FakeGeocoder
    : IReverseGeocoder
{
    public Task<string?> ResolveAsync(
        double lat, double lon,
        CancellationToken ct)
    {
        // Deterministisch, kein Netzwerk
        var name = $"Ort@{lat:F2}/{lon:F2}";
        return Task.FromResult<string?>(name);
    }
}
Das Interface IReverseGeocoder macht die Austauschbarkeit möglich — ein zentrales Prinzip der testbaren Architektur.

Was testen wir?

  • ✓
    ParseLocation Korrekte Feld-Priorisierung (village vor town vor city)
  • ✓
    Cache-Treffer Zweite Abfrage gleicher Koordinaten → kein HTTP-Request
  • ✓
    Cache-Persistenz Neues Cache-Objekt lädt gespeicherte Werte aus JSON-Datei
  • ✓
    Abbruch OperationCanceledException → graceful exit, kein Crash

Zusammenfassung

Algorithmus

  1. EXIF-GPS aus JPEG lesen
  2. Cache prüfen (4 Dezimalstellen)
  3. Rate-Limiter: 1 Request/s
  4. HTTP GET an Nominatim
  5. JSON parsen, Feld-Kaskade
  6. Ergebnis cachen & zurückgeben
