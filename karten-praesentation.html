ALGORITHMUS 1

Web-Mercator-Projektion

Problem

Die Erde ist eine Kugel. Karten sind flach.
Wie rechnen wir GPS-Koordinaten in Bildpixel um?

Schritt 1 — Längengrad → X

Longitude geht von −180° bis +180°.
Einfache lineare Abbildung:

x = (lon + 180) / 360 × Gesamtbreite

Schritt 2 — Breitengrad → Y (nicht linear!)

Latitude braucht die Mercator-Formel (logarithmisch), damit Landmassen proportional bleiben:

latRad = lat × π / 180
y = (1 − ln(tan(latRad) + 1/cos(latRad)) / π) / 2 × Gesamthöhe

Zoom-Level

Gesamtbreite/-höhe = 2zoom × 256 Pixel.
Bei Zoom 10: 256 × 1024 = 262 144 px breite Weltkarte.

public static (double X, double Y) LatLonToGlobalPixel(
    double lat, double lon, int zoom)
{
    double scale  = Math.Pow(2, zoom) * TileSize;

    // Longitude → X (linear)
    double x = (lon + 180.0) / 360.0 * scale;

    // Latitude → Y (Mercator-Formel)
    double latRad = lat * Math.PI / 180.0;
    double y = (1.0 -
        Math.Log(Math.Tan(latRad) +
               1.0 / Math.Cos(latRad)) / Math.PI
    ) / 2.0 * scale;

    return (x, y);
}

// Globale Pixel → Tile-Index (ganzzahlig)
public static (int, int) GlobalPixelToTile(double x, double y)
    => ((int)(x / TileSize), (int)(y / TileSize));

Warum Mercator?

Winkeltreu: Straßenverläufe und Formen wirken lokal korrekt. Skalierungsfehler nimmt zu den Polen zu — für Mitteleuropa gut geeignet.

TileSize = 256

Standard weltweit: OpenStreetMap, Google Maps, MapTiler — alle verwenden 256×256 px Kacheln.

Welche Tiles werden gebraucht?

Aus Bounding-Box (Hüllrechteck aller GPS-Punkte) + Rand-Padding berechnen wir den Tile-Bereich:

var (tileXMin, tileYMin) =
    GlobalPixelToTile(rangeMinGx, rangeMinGy);
var (tileXMax, tileYMax) =
    GlobalPixelToTile(rangeMaxGx, rangeMaxGy);

// Alle Tiles im Rechteck:
for (int tx = tileXMin; tx <= tileXMax; tx++)
  for (int ty = tileYMin; ty <= tileYMax; ty++)
    tiles.Add(new MapTile(zoom, tx, ty));

Tile-URL — zwei Anbieter

MapTiler (API-Key): hochwertige Karten
OpenStreetMap (kostenlos): Fallback ohne Key
Beide liefern identisch adressierte 256×256 PNGs.

ALGORITHMUS 3

Mosaik-Komposition & Zuschnitt

Schritt 1 — Tiles parallel herunterladen

var semaphore = new SemaphoreSlim(maxParallel);

await Task.WhenAll(tiles.Select(async tile =>
{
    await semaphore.WaitAsync(ct);
    try {
        var data = await _cache.TryGetAsync(key, ct)
                   ?? await DownloadAndCacheAsync(tile, ct);
        lock (tileData) tileData[tile] = data;
    } finally { semaphore.Release(); }
}));

Task.WhenAll = alle Tiles gleichzeitig (bis max. 4 parallel)
SemaphoreSlim = begrenzt gleichzeitige HTTP-Anfragen

Tile-Cache (FileTileCache)

Jeder heruntergeladene Tile wird als Datei gespeichert (.tile-cache/osm/z/x/y.png). Atomares Schreiben via Temp-Datei + Move verhindert korrupte Caches.

ALGORITHMUS 4

Stopp-Erkennung

Idee

Wenn zwischen zwei ausgewählten Fotos eine Zeitlücke größer als die Schwelle (z. B. 30 Minuten) liegt, hat der Reisende eine Pause gemacht — ein Stopp!

Zeitstrahl-Beispiel (Schwelle = 30 min)

public IReadOnlyList<StopPoint> DetectStops(
    IReadOnlyList<Photo> selectedSorted,
    TimeSpan threshold,
    bool addFinalSummary)
{
    var gps = selectedSorted
        .Where(p => p.Latitude.HasValue
                 && p.DateTime.HasValue)
        .ToList();

    var result = new List<StopPoint>();

    for (int i = 0; i < gps.Count - 1; i++)
    {
        var gap = gps[i+1].DateTime!.Value
                - gps[i].DateTime!.Value;

        if (gap > threshold)
            result.Add(new StopPoint(
                gps[i].DateTime!.Value.AddSeconds(1),
                gps[i].Latitude!.Value,
                gps[i].Longitude!.Value));
    }

    // Letztes Foto = Zusammenfassungskarte der ganzen Reise
    if (addFinalSummary && gps.Count > 0)
        result.Add(new StopPoint(
            gps[^1].DateTime!.Value.AddSeconds(1),
            gps[^1].Latitude!.Value,
            gps[^1].Longitude!.Value));

    return result;
}

Wichtig: nur Selected-Fotos!

Die Stopp-Erkennung arbeitet nur mit ausgewählten Fotos — das ist die narrative Reiseauswahl des Nutzers. Nicht ausgewählte Fotos beeinflussen die Stopp-Logik nicht.

ALGORITHMUS 5

Polyline — Route zeichnen

Datenquelle der Route: ALLE GPS-Fotos

Die Polyline zeigt den realen Streckenverlauf — durch alle GPS-Fotos (auch nicht ausgewählte). Zeitbasierter Filter: alles bis zum Stopp-Zeitstempel.

// Für jeden Stopp individuell:
var routePixels = allGpsPhotos
    .Where(p => p.DateTime!.Value <= stop.Timestamp)
    .Select(p => ToImg(p.Latitude!.Value,
                          p.Longitude!.Value))
    .ToList();

var simplified = SimplifyByPixelDistance(routePixels);

if (simplified.Length >= 2)
{
    img.Mutate(ctx => {
        // Weiße Kontur (8 px) + blaue Linie (5 px)
        ctx.DrawLine(new SolidPen(Color.White, 8f),
                       simplified);
        ctx.DrawLine(new SolidPen(Color.ParseHex("4285F4"), 5f),
                       simplified);
    });
}

Warum zeitbasiert statt indexbasiert?

Stopps kommen aus den Selected-Fotos, Route aus allen Fotos — beide Listen haben verschiedene Indizes. Zeit ist der gemeinsame Nenner.

Punktvereinfachung — SimplifyByPixelDistance

200 Foto-Punkte erzeugen meist 150+ Pixel-Punkte, von denen viele weniger als 2 px auseinander liegen. Der Vereinfachungs-Algorithmus filtert überflüssige Punkte:

private static PointF[] SimplifyByPixelDistance(
    IReadOnlyList<PointF> points,
    float minDistancePx = 2.0f)
{
    var result = new List<PointF> { points[0] };

    for (int i = 1; i < points.Count - 1; i++)
    {
        var last = result[^1];
        float dx = points[i].X - last.X;
        float dy = points[i].Y - last.Y;
        // Pythagoras (ohne Wurzel, zum Vergleich reicht dx²+dy²)
        if (dx*dx + dy*dy >= minDistancePx * minDistancePx)
            result.Add(points[i]);
    }

    result.Add(points[^1]); // letzten immer übernehmen!
    return result.ToArray();
}

Effekt

Typisch werden 60–80 % der Punkte entfernt, ohne sichtbaren Qualitätsverlust. Der letzte Punkt (= Stopp) wird immer beibehalten, damit die Linie exakt am Marker endet.

// Kreis zeichnen mit ImageSharp.Drawing
private static void DrawDot(
    Image<Rgba32> image, PointF center,
    float radius, Color fill,
    Color border, float borderWidth)
{
    var circle = new EllipsePolygon(center,
        new SizeF(radius * 2, radius * 2));

    image.Mutate(ctx => {
        ctx.Fill(new SolidBrush(fill), circle);
        ctx.Draw(new SolidPen(border, borderWidth),
                  circle);
    });
}

// Schatten-Effekt durch verschobene Halbkreise
private static void DrawShadowedDot(...) {
    var shadow = new Color(new Rgba32(0,0,0,60));
    for (int i = 1; i <= 2; i++) {
        var sc = new PointF(center.X, center.Y + i*2);
        image.Mutate(ctx => ctx.Fill(new SolidBrush(shadow),
            new EllipsePolygon(sc, ...)));
    }
    DrawDot(image, center, radius, fill, border, bw);
}

QUALITÄT

Testbarkeit durch Dependency Injection

Problem: Echter Netzwerkzugriff in Tests

Unit-Tests sollen schnell, stabil und offline funktionieren. Echte Tile-Downloads würden Tests langsam, instabil und netzwerkabhängig machen.

Lösung: Interface + FakeTileSource

Der Renderer kennt ITileSource nur über ein Interface. Im Test wird eine FakeTileSource übergeben, die sofort gefärbte 256×256-Bitmaps zurückgibt:

sealed class FakeTileSource : ITileSource
{
    public string ProviderId      => "fake";
    public string AttributionText => "Fake tiles";

    public Task<byte[]> GetTileAsync(MapTile tile, ...)
    {
        // Einfarbige 256×256-PNG, Farbe variiert pro Tile
        byte r = (byte)((tile.X * 37) & 0xFF);
        byte g = (byte)((tile.Y * 71) & 0xFF);
        using var img = new Image<Rgba32>(256, 256,
            new Rgba32(r, g, 180));
        using var ms = new MemoryStream();
        img.SaveAsPng(ms);
        return Task.FromResult(ms.ToArray());
    }
}

Wichtigste Design-Entscheidung

Trennung von Route und Stopps: Polyline und Bounding-Box kommen aus allen GPS-Fotos (reale Spur), während Stopp-Erkennung und Marker nur auf ausgewählten Fotos basieren (narrative Auswahl). Dadurch wird die Karte sowohl technisch präzise als auch redaktionell kontrolliert.
